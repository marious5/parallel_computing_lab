### 1. 实验目的

本实验旨在通过CUDA平台实现并行的通用矩阵乘法 。在正确完成随机生成的 $m \times n$ 矩阵 $A$ 与 $n \times k$ 矩阵 $B$ 相乘求得矩阵 $C$ 的基础上，深入对比不同矩阵规模（取值范围在[128, 2048]之间）下的耗时差异。同时，探究不同线程块大小如何影响硬件流多处理器的占用率，以及全局内存与共享内存的分块访存策略、数据与任务的划分方式对程序最终计算时间 $t$ 的实际影响。

### 2. 实验过程和核心代码

**2.1 实验环境准备** 实验在国家超级计算广州中心（天河二号）的计算节点上进行。编写好源程序后，使用 `nvcc` 编译器加 `-O3` 优化选项进行编译，并通过批处理脚本依次扫描不同矩阵规模与线程块大小，从而捕获多组执行数据。

**2.2 核心代码实现与优化** 程序中采用二维网格（Grid）和二维线程块（Block）对数据进行划分，每个线程计算矩阵 $C$ 中的一个元素。实验实现了两种内核对比：

**1. 基础版本 (Naive Global Memory)：**

直接按行列坐标从全局内存读取矩阵 $A$ 和 $B$ 的元素进行乘加运算。由于全局访存延迟较高，对带宽占用大。

```C++
__global__ void matrixMulNaive(float *A, float *B, float *C, int m, int n, int k) {
    int row = blockIdx.y * blockDim.y + threadIdx.y;
    int col = blockIdx.x * blockDim.x + threadIdx.x;
    if (row < m && col < k) {
        float sum = 0.0f;
        for (int i = 0; i < n; ++i) {
            sum += A[row * n + i] * B[i * k + col];
        }
        C[row * k + col] = sum;
    }
}
```

**2. 共享内存分块版本 (Shared Memory Tiling)：** 利用片上共享内存将大矩阵乘法划分为大小为 `TILE_SIZE` 的小数据块。每个Block内的线程协同将全局内存中的数据载入共享内存，调用 `__syncthreads()` 同步后，在低延迟的共享内存中完成局部乘法 。

```C++
// 共享内存分块核心逻辑片段
for (int t = 0; t < numTiles; ++t) {
    // 协同加载至共享内存 s_A 和 s_B... (越界处理省略)
    __syncthreads();
    for (int i = 0; i < TILE_SIZE; ++i) {
        sum += s_A[ty * TILE_SIZE + i] * s_B[i * TILE_SIZE + tx];
    }
    __syncthreads();
}
```

### 3. 实验结果

通过脚本自动化测试，我们收集了输出矩阵乘法时间 $t$ 的核心数据，将其整理为以下分析图表。

**测试一：矩阵规模对性能的影响 (固定线程块 16x16)**

| **矩阵规模 (M=N=K)** | **Naive 耗时 (ms)** | **Shared 耗时 (ms)** |
| -------------------- | ------------------- | -------------------- |
| 128                  | 0.0109              | 0.0121               |
| 512                  | 0.1280              | 0.1461               |
| 1024                 | 0.9219              | 1.0371               |
| 2048                 | 7.2379              | 8.1331               |

随着矩阵规模倍增，计算量呈立方级增长，但得益于GPU的大规模并发，耗时增长维持在合理区间。出乎意料的是，Shared版本在此次测试中比Naive版本慢约10%。这通常是因为当前的Tiling逻辑引入了额外的指令开销和同步等待，而在基础计算强度下，L1/L2 Cache已经对Naive版本的全局访存起到了很好的缓冲作用。

**测试二：线程块大小对性能的影响 (固定规模 2048x2048)**

| **线程块大小 (BlockSize)** | **Naive 耗时 (ms)** | **Shared 耗时 (ms)** |
| -------------------------- | ------------------- | -------------------- |
| 4x4 (16线程)               | 18.459              | 47.319               |
| 8x8 (64线程)               | 8.328               | 9.271                |
| 16x16 (256线程)            | **6.874**           | **7.692**            |
| 32x32 (1024线程)           | 7.143               | 8.617                |

数据显示，线程块大小极大地影响了硬件资源的占用率 。当BlockSize过小（如4x4）时，无法分配足够的Warp来隐藏指令和访存延迟，导致耗时激增（Shared版飙升至47ms）。而在16x16时性能达到最优。当扩大至32x32时，由于寄存器资源压力增加，反而导致活跃的Block数量下降，性能再次回落。

之前在完成基于OpenMP和MPI的高性能计算实验时，我主要关注CPU端的多核协作与节点间通信。这次转向CUDA架构，结合我之前在底层系统和C++内核开发中积累的经验，让我对GPU的线程层次划分（Grid/Block/Thread）和底层访存机制有了非常直观的认识。

实验中最让我印象深刻的问题是“共享内存版本耗时反常”。经过排查和分析，我意识到理论上的优化必须结合实际的硬件特性——基础的共享内存策略若没有处理好Bank Conflict或充分展开循环，其显式的同步开销完全可能抵消甚至拖垮缓存带来的收益。同时，通过调整Block大小从4x4到16x16，我深刻体会到了“通过高并发掩盖延迟”这一GPU核心设计哲学的威力。这对我后续进行系统级的性能瓶颈定位大有裨益。
