### 1. 实验目的（200字以内）

本次实验旨在深入理解并行计算中循环并发的底层机制。通过剥离 OpenMP 的高级抽象，利用底层 Pthreads 库从零手工构建一个类似 `omp_parallel_for` 的并行调度构造。在此基础上，将该自定义并发构造应用于经典的二维网格热传导模拟（Heated Plate）问题中，通过控制变量法对比不同线程数下的并行效率，分析由于线程创建、销毁以及任务调度带来的系统级开销，从而加深对并行程序性能调优及底层任务分发逻辑的理解。

### 2. 实验过程和核心代码（600字以内，图文并茂）

#### 2.1 实验环境准备

本次实验在 Linux 环境下完成（基于 HPC 容器节点），编译器为 GCC，利用 `-lpthread` 链接 POSIX 线程库，并生成 `.so` 动态链接库以供主程序调用。节点最高可用核心数为 16 线程。

#### 2.2 核心代码实现与优化

**（1）动态调度机制实现**

在 `parallel_for` 的实现中，我们采用了静态块调度（Static Block Scheduling）策略。核心思想是根据 `start`、`end` 和 `num_threads` 计算出总迭代次数，并将其尽可能均匀地切分为若干个 Chunk 分发给各个线程。若存在余数，则将多出的任务优先分配给前面的线程，确保负载均衡。

C

```
// 核心任务切分逻辑
int total_iters = (end - start + inc - 1) / inc; 
int chunk = total_iters / actual_threads;
int remainder = total_iters % actual_threads;

for (int i = 0; i < actual_threads; i++) {
    t_data[i].start = current_start;
    int current_chunk = chunk + (i < remainder ? 1 : 0);
    t_data[i].end = current_start + current_chunk * inc;
    // ... 绑定 functor 和参数并执行 pthread_create
}
```

**（2）无锁规约优化（Reduction）**

在改写 Heated Plate 应用时，最棘手的是最大误差 `diff` 的求取。OpenMP 中使用了 `#pragma omp critical`，这在底层会引入锁竞争。为了追求极限性能，我采用了一种“空间换时间”的无锁策略：为每一行分配一个私有的 `row_diffs[M]` 数组。每个线程计算时只将最大误差写入自己负责行的对应索引中，从而完全避免了数据竞争。循环结束后，再由主线程串行扫描该数组完成最终的规约操作。

### 3. 实验结果（500字以内，图文并茂）

矩阵乘法测试正确输出预期值，证明了并行拆分逻辑的正确性。随后进行的 Heated Plate 热传导模拟测试中，程序设定的收敛阈值为 0.001，各版本的性能数据如下表所示：

| **并行实现版本** | **线程数** | **迭代次数** | **最终计算误差** | **Wallclock 耗时 (秒)** |
| ---------------- | ---------- | ------------ | ---------------- | ----------------------- |
| 原版 OpenMP      | 16         | 16955        | 0.001000         | 2373.900026             |
| 自定义 Pthreads  | 1          | 16955        | 0.001000         | 22.112443               |
| 自定义 Pthreads  | 2          | 16955        | 0.001000         | **20.991637**           |
| 自定义 Pthreads  | 4          | 16955        | 0.001000         | 26.693963               |
| 自定义 Pthreads  | 8          | 16955        | 0.001000         | 45.404827               |

**结果分析：**

1. **算法正确性**：所有版本的迭代次数均精确稳定在 16955 次，且最终误差完全一致，验证了 Pthreads 改造版本算法的绝对正确。
2. **多线程反常性能损耗**：自定义的 Pthreads 仅在 2 线程时取得了微弱加速比（20.99秒），随后耗时随线程数增加而显著劣化的现象（8线程达到45.4秒）。这主要是因为 `parallel_for` 位于 `while` 的密集迭代（近1.7万次）内部，每次循环都要频繁触发底层的 `pthread_create` 和 `pthread_join`。对于 500x500 这种小规模网格，系统调用的开销远超出了并行计算带来的收益。
3. **OpenMP 异常耗时**：原版 OpenMP 使用 16 线程耗时高达 2373.9 秒，推测是由于该实验节点（如 Docker 容器）受到 CPU 调度配额限制，16 线程在底层引发了灾难性的上下文切换风暴（Context Switch Storm）或自旋锁过度竞争。

### 4. 实验感想（200字以内）

这次实验的结果非常具有戏剧性，也让我收获颇丰。在以往编写 OS 内核及处理底层架构时，我深知系统调用的昂贵开销，而本次实验直观地量化了这一代价：在近两万次的高频迭代中，频繁地进行线程创建与销毁，反而成为了性能的致命瓶颈。这警示我在未来的高并发场景（如高帧率 UI 渲染或服务端通信）中，不能盲目迷信“多线程即高效率”。合理使用线程池进行复用、精细把控任务粒度，以及尽可能消除锁竞争，才是发挥多核硬件极限性能的核心关键。
